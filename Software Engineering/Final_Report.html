1 Final Project Description

Purpose

DragRace.io is a web-based quarter-mile drag-race simulator designed for automotive and motorcycle enthusiasts who want to compare real-world vehicle performance in an engaging, visual format. The system allows users to browse a curated catalog of 149 real vehicles — spanning supercars, sports sedans, muscle cars, electric vehicles, and motorcycles — select any two for a head-to-head matchup, and watch a 3D race animation that reflects their actual quarter-mile times.

Problem

Enthusiasts frequently debate which vehicles are fastest, but existing resources (car review sites, YouTube comparisons) lack an interactive, standardized way to pit any two vehicles against each other using consistent data. DragRace.io solves this by providing a single platform where users can instantly compare any pair from a verified performance database.

Customer & Users

The primary users are casual and dedicated car and motorcycle enthusiasts who enjoy quarter-mile performance comparisons. The system requires no account creation, making it immediately accessible to any visitor.

Delivered Features

The final system provides five interconnected pages:

  1. Home — Browse 149 vehicles in a searchable, filterable grid with real-time debounced search. Select two vehicles (A and B) for a head-to-head matchup.
  2. Race Results — Watch a Three.js 3D bird's-eye race animation with proper pacing based on actual quarter-mile times, then view the winner, both times, and the margin of victory.
  3. Vehicle Detail — Inspect full specifications for any vehicle, including make, model, trim, year, type, quarter-mile time, and data source.
  4. Leaderboard — View all vehicles ranked by quarter-mile time with gold/silver/bronze podium highlighting and type filtering.
  5. Race History — Review all past race results with human-readable vehicle names, winner identification, time margins, and timestamps.

Cross-cutting features include dark/light theme toggling with persistence, comprehensive keyboard accessibility, responsive layout (desktop/tablet/mobile), API rate limiting, input validation, and server-side response caching.

Evolution from Earlier Iterations

Sprint 01 delivered the core browse-select-race-results flow as a minimal viable product. Sprint 02 added visual polish (Three.js race animation), security hardening (rate limiting, restricted CORS), and accessibility (ARIA labels, keyboard navigation, prefers-reduced-motion support). Sprint 03 expanded the feature set with vehicle details, leaderboard, race history, caching, theme toggle, and a 45% increase in the vehicle catalog (103 → 149). Each sprint delivered a working, tested increment that built on top of the previous one.

3 Design Document

3.1 System Architecture

DragRace.io uses a two-tier client-server architecture:

  • Frontend (React SPA): A single-page application built with Vite, React 19, and TypeScript. It communicates with the backend exclusively through RESTful HTTP calls using the Fetch API. Client-side routing is managed by react-router-dom. The Three.js library powers the 3D race animation with GLTF model loading.
  • Backend (FastAPI REST API): A Python FastAPI application exposing five REST endpoints. It uses SQLAlchemy ORM for database access, Pydantic for request/response validation, slowapi for rate limiting, and cachetools for server-side response caching.
  • Database (SQLite): Two tables — vehicles (149 records seeded from CSV) and race_history (automatically populated when races occur). SQLite was chosen for zero-configuration local development; the ORM abstraction allows straightforward migration to PostgreSQL for production.

3.3 Database Schema

-- Table: vehicles
CREATE TABLE vehicles (
    id                   INTEGER PRIMARY KEY AUTOINCREMENT,
    type                 VARCHAR NOT NULL,     -- "car" | "motorcycle"
    make                 VARCHAR NOT NULL,
    model                VARCHAR NOT NULL,
    trim                 VARCHAR DEFAULT '',
    year                 INTEGER NOT NULL,
    quarter_mile_time_s  FLOAT NOT NULL,
    source               VARCHAR DEFAULT ''
);
CREATE INDEX ix_vehicles_make ON vehicles (make);
CREATE INDEX ix_vehicles_model ON vehicles (model);
CREATE INDEX ix_vehicles_year ON vehicles (year);
CREATE INDEX ix_vehicles_quarter_mile ON vehicles (quarter_mile_time_s);

-- Table: race_history
CREATE TABLE race_history (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    vehicle_a_id     INTEGER NOT NULL REFERENCES vehicles(id),
    vehicle_b_id     INTEGER NOT NULL REFERENCES vehicles(id),
    winner           VARCHAR NOT NULL,  -- "A" | "B" | "Tie"
    vehicle_a_time_s FLOAT NOT NULL,
    vehicle_b_time_s FLOAT NOT NULL,
    diff_s           FLOAT NOT NULL,
    created_at       DATETIME DEFAULT CURRENT_TIMESTAMP
);

4.3 Feature Walkthrough

Browse & Search Vehicles

  1. The Home page displays a grid of vehicle cards showing make, model, trim, year, type, and quarter-mile time.
  2. Type in the search bar (e.g., "corvette") — results filter after a 300ms debounce.
  3. Use the type dropdown to filter by "Cars" or "Motorcycles."
  4. Click a vehicle's title to navigate to its detail page.

Set Up a Race

  1. Click "Set as A" on any vehicle card to select it as Vehicle A.
  2. Click "Set as B" on a different vehicle to select it as Vehicle B.
  3. Selected vehicles appear in the matchup bar at the top.
  4. Click "🚀 Race!" to start the matchup.

Watch the Race & View Results

  1. A Three.js 3D bird's-eye race animation plays (~8 seconds). The faster vehicle crosses first.
  2. Click "⏩ Skip Animation" to jump directly to results.
  3. The Results panel shows: winner banner, side-by-side comparison, and time margin.
  4. A simulation disclaimer appears below the results.
  5. Click "↩ New Matchup" to return to the Home page.

Explore the Leaderboard

  1. Click "🏆 Leaderboard" in the navigation bar.
  2. Vehicles are ranked fastest to slowest with gold, silver, and bronze highlighting.
  3. Use the type filter to see "Cars Only" or "Motorcycles Only."

View Race History

  1. Click "📜 History" in the navigation bar.
  2. Each past race shows the A vs B matchup with the winner highlighted in green.
  3. The most recent race appears at the top.

Toggle Theme

  1. Click the "☀️ Light" / "🌙 Dark" button in the top-right corner.
  2. The theme persists across page reloads (stored in localStorage).

5 Summary of Testing

5.1 Methodology

Testing was conducted at four levels throughout the project:

  1. Automated Unit & Integration Tests (pytest): 39 automated tests using FastAPI's TestClient with an isolated SQLite test database. Tests cover all five API endpoints, input validation, error handling, race prediction logic, leaderboard ordering, history persistence, caching behavior, and cross-module consistency.
  2. Manual Functional Testing: Systematic manual test cases executed against the running application, covering full user workflows, edge cases, UI states, and responsive layout behavior. The Sprint 03 regression checklist contains 35 items.
  3. Product Owner Acceptance Testing: The Product Owner reviewed each user story against its acceptance criteria at the end of every sprint and formally accepted or rejected stories.
  4. Automated Demo Testing (Playwright): A Playwright script (demo.js) automates the full application workflow for the demo video, verifying theme persistence, cache behavior (X-Cache headers), navigation, search, race animation, and the simulation disclaimer.

6 Ethical Dimensions Report

6.1 Privacy

DragRace.io collects no personal data. There is no user registration, no login, no cookies (beyond browser defaults), and no analytics tracking. The only data persisted is the vehicle catalog (pre-loaded) and race history records (containing only vehicle IDs, times, and timestamps — no user identifiers).

This was an intentional design decision: by not building user accounts, we eliminated the entire class of risks associated with password storage, data breaches, account enumeration, and GDPR/PIPEDA compliance obligations. If authentication were added in the future, it would introduce responsibilities around secure password hashing, session management, and data retention policies that would need to be carefully addressed.

6.2 Security

Despite being a local-only prototype, the system implements several security measures:

  • CORS Restriction (SEC-01): The open CORS policy (*) from Sprint 01 was replaced in Sprint 02 with an explicit allowlist (localhost:5173, localhost:4173). This prevents unauthorized cross-origin requests from arbitrary domains.
  • Rate Limiting: All five endpoints are rate-limited to 60 requests per minute per IP via slowapi. This mitigates denial-of-service and brute-force abuse.
  • Input Validation: FastAPI's Pydantic validation rejects malformed requests at the API boundary. Search strings are capped at 100 characters. Year filters are constrained to 1900–2030. Pagination enforces non-negative offsets and a maximum limit of 100.
  • SQL Injection Prevention: All database queries use SQLAlchemy's parameterized ORM queries — no raw SQL concatenation.

Remaining risks: The application runs over HTTP (not HTTPS) in development. A production deployment would require TLS encryption. The lack of authentication means any user can access the full API.

6.3 Accessibility

Accessibility was a first-class concern starting in Sprint 02:

  • ARIA Labels: All interactive elements carry descriptive aria-label attributes. The matchup bar, vehicle grid, and results section use appropriate role attributes.
  • Keyboard Navigation: All controls are reachable via Tab. Focus-visible outlines (purple, 2px with offset) appear on all focusable elements using the :focus-visible pattern.
  • Reduced Motion: The CSS includes a prefers-reduced-motion: reduce media query that disables all animations. The Three.js race animation explicitly checks this preference and skips directly to results.
  • Skip Link: A hidden skip-to-content link appears on keyboard focus for screen reader users.
  • Semantic HTML: Headings follow a logical hierarchy. Error states use role="alert" for screen-reader announcement.

Areas for improvement: We did not conduct a formal WCAG 2.1 AA audit. Color contrast ratios in the dark glassmorphism theme may not meet the 4.5:1 minimum for small text. A production deployment should include an axe-core audit and screen reader testing (VoiceOver, NVDA).

6.4 Fairness & Bias in Data

The vehicle catalog is sourced from published quarter-mile test data from reputable automotive publications (MotorTrend, Car and Driver, Road and Track, Cycle World, Motorcyclist). Each vehicle record includes a source field attributing the data origin.

Potential bias concerns:

  • The catalog over-represents premium and performance vehicles relative to mainstream cars. A user might assume the leaderboard represents all vehicles, when it is a curated subset.
  • Quarter-mile times vary based on test conditions. Our data uses published results from professional testers, but real-world results will differ.

Mitigations implemented:

  • The results page includes a prominent disclaimer: "DragRace.io is a simulation tool for educational purposes only. Vehicle performance data is estimated and does not reflect real-world conditions."
  • The source field on every vehicle and the Vehicle Detail page lets users trace data provenance.

6.5 Safety & Responsible Use

There is an inherent risk that a "race" application could encourage dangerous driving behavior. We addressed this through:

  1. The LEGAL-01 disclaimer explicitly states the tool is for educational and entertainment purposes only.
  2. The application simulates results using published test data — it does not connect to real vehicles or encourage real-world racing.
  3. The system makes no claims about vehicle safety, handling, or real-world suitability.

6.6 Summary

DragRace.io adopts a "minimal footprint" approach to ethics: by collecting no personal data, restricting API access, implementing accessibility features, attributing all data sources, and including safety disclaimers, the system addresses the primary ethical dimensions relevant to its scope. The most important future improvement would be a comprehensive WCAG accessibility audit and color contrast remediation.

7 Final Reflection

What Is Fully Completed

All 19 planned user stories were implemented, tested, and accepted by the Product Owner. The system delivers the complete user flow from vehicle browsing through race simulation to history review, supported by 39 passing automated tests, a 35-item regression checklist, and three sprint demo videos. Non-functional requirements — security hardening, accessibility, responsive design, and performance caching — are all implemented with concrete, verifiable evidence in the codebase.

Biggest Technical Challenge

The most complex feature was the Three.js race animation (RACE-02/RACE-03). Building a real-time 3D scene that properly reflects relative vehicle performance required careful animation timing calculations — normalizing quarter-mile times to an 8-second visual duration while maintaining the correct proportional gap between the two vehicles. We also had to handle the GLTF model loading pipeline with a robust fallback path when GLB assets are unavailable, and implement ease-in-out physics to make the animation feel natural. Integrating this with React's lifecycle (cleanup, resize handlers, reduced-motion checks) added additional complexity.

A secondary challenge was server-side caching that avoids SQLAlchemy's DetachedInstanceError. We solved this by serializing ORM objects to Pydantic models before inserting them into the TTL cache, ensuring cached responses remain valid after the database session closes.

What We Would Improve with More Time

  1. User authentication and per-user history — allowing users to save their own matchup history.
  2. Cloud deployment — deploying the backend to Render and the frontend to Vercel for public access.
  3. WCAG accessibility audit — running axe-core and conducting screen-reader testing.
  4. Frontend automated tests — adding React Testing Library and Playwright E2E tests.
  5. Expanded catalog — growing beyond 149 vehicles using community-sourced data.

The most important lesson from this project was the value of building a vertical slice first. By getting the full browse → race → results flow working in Sprint 01, we proved the architecture early and could confidently layer features on top in subsequent sprints without architectural rework.
